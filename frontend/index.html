System Analysis: Trust Model, Centralisation Risks & Scalability Argument
▼

1. Architecture Overview

User Wallet │ submit intent (type, amount, token, recipient, preference) ▼ IntentRegistry.sol ←─ stores intent ID & routing decision hash on-chain │ ▼ Off-chain Intent Router (this server) ├─ scores RollupA (ArbiNova) — fee, latency, success probability ├─ scores RollupB (OptiSwift) — fee, latency, success probability └─ scores RollupC (ZkRapid) — fee, latency, success probability │ └─→ selects winner → MockRollup.executeIntent() │ ▼ IntentRegistry.recordExecution()

2. Trust Assumptions

The routing engine is off-chain — this is the primary trust assumption in the system. Users must trust that the router:

  • Selects the genuinely best rollup for their preference (not the one paying the router a kickback).
  • Reports accurate fee and latency data (not spoofed).
  • Does not censor specific users or intents.

Mitigations implemented in this prototype:

  • On-chain reason hash: The IntentRegistry contract supports storing a keccak256 hash of the routing rationale, letting any observer verify the router's stated reason after the fact. (This demo runs in simulation mode — the contract is deployed and the interface is wired; live submission is a straightforward extension.)
  • Open routing logic: The scoring algorithm is open-source — anyone can re-run it with the same rollup snapshots and verify the decision.
  • User-controlled preference: Users express a routing preference (cheapest / fastest / balanced) that constrains the router's freedom.

3. Centralisation Risks

A single router is a single point of failure and control. Risks:

  • Liveness: If the router goes offline, intents cannot be routed. Mitigation: multiple competing routers, user can fall back to direct submission.
  • Censorship: Router can refuse to route certain addresses. Mitigation: permissionless fallback path directly to any rollup.
  • MEV extraction: Router sees intent content before execution — can front-run. Mitigation: commit-reveal scheme or private mempool.

4. Scalability Argument

Without a routing layer, users face the following friction in a multi-rollup world:

  • Must manually compare fees across N rollups before every transaction.
  • Must bridge assets to the cheapest rollup before executing.
  • As N grows (hundreds of appchains), the UX becomes unusable.

The intent router solves this by:

  • Abstracting chain choice: Users express what they want, not where to do it.
  • Automated routing: The router can query N rollups in parallel in milliseconds.
  • Fee savings at scale: In high-congestion scenarios, routing to the least congested chain reduces fees by up to 6× (see benchmark analysis).
  • Batching: Multiple intents with similar parameters can be batched into one on-chain transaction, amortising fixed calldata cost across N users.

5. When Routing Helps Most

Scenario A — Uniform congestion (all rollups ~50%): Fee spread: 0.75 → 0.82 → 1.70 gwei Saving from routing: ~0.95 gwei per intent (moderate benefit) Scenario B — One congested rollup (rollupA ~90%, others ~20%): Fee spread: 0.95 → 0.63 → 0.87 gwei Router avoids rollupA, saves ~0.32 gwei (routing critical) Scenario C — All congested equally (~90%): Fee spread: 0.95 → 2.90 → 7.89 gwei ZK rollup most expensive; routing saves 6.94 gwei (huge benefit) Scenario D — All lightly loaded (~5%): Fee spread: 0.50 → 1.20 → 3.00 gwei Routing overhead (~1ms) vs. fixed fee gap (always beneficial)

Two distinct patterns dominate the benefit analysis:

  • Largest absolute savings → uniform high congestion (Scenario C). When all chains are equally stressed, ZK proof overhead causes the fee gap to widen most — the router captures the full spread by routing away from the most expensive chain.
  • Highest adaptive value → asymmetric congestion (Scenario B). When one chain spikes while others stay low, a fixed-chain user gets stuck paying the spike; the router detects the shift and re-routes automatically. This is the most common real-world pattern, as rollups serve different user segments.

Both scenarios justify a routing layer; the dominant benefit depends on current network conditions.

6. ZK vs. Optimistic Rollups

This prototype includes both rollup types, reflecting the real-world ecosystem:

  • Optimistic rollups (ArbiNova, OptiSwift): lower base cost but 7-day withdrawal challenge window. Suitable for non-urgent DeFi, transfers.
  • ZK rollups (ZkRapid): higher proof generation cost but instant cryptographic finality. Suitable for high-value swaps requiring settlement guarantees.

The routing engine correctly prefers optimistic rollups for "cheapest" intents and ZK rollups for "fastest" intents, matching the theoretical optimal for each type.
